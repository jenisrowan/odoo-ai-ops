### Infrastructure Notes (Updated)
* [ AWS WAF ] Security layer protecting CloudFront with Bot Control, Rate Limiting, and IP Reputation.
* [ Load Balancer ] is a single ALB (aws_lb.main) spanning both public subnets for HA.
* [ Regional NAT GW ] provides secure egress for external API calls only (Anthropic, Shopify, Slack).
* [ VPC Endpoints ] Gateway VPC Endpoint for S3 (free) provides private connectivity without NAT data processing charges. Interface VPC Endpoint for SQS (Amazon PrivateLink) allows secure and private polling of the SQS queue without traversing the NAT Gateway. ECS and RDS are in the same VPC and communicate directly without endpoints.
* [ Odoo Service ] Odoo connects to EFS Mounting target for file storage and PgBouncer for database pooling.
* [ PgBouncer HA ] It uses ECS Service Connect (pgbouncer.odoo.local) for discovery.
* [ EFS Mount Target ] are two mount targets of one shared EFS file system (aws_efs_file_system.odoo).
* [ RDS Postgres ] is a db.m6g.xlarge instance with multi_az=true; AWS manages the standby replica automatically.

* **[ Webhook Ingestion ]** Webhooks are routed by CloudFront to **AWS API Gateway**, which invokes a **Lambda proxy integration** (the Webhook Verify+Ingest Lambda). That single Lambda verifies the HMAC signature, answers Slack's synchronous URL-verification challenge, and only on success, enqueues the normalized payload (`{source, topic, payload}`) into **Amazon SQS**. Fusing verification and enqueue in one proxy Lambda is required because an HTTP API Lambda *authorizer* cannot emit the Slack challenge response body; it guarantees no unverified message ever reaches the queue.
* **[ Load Balancer Routing ]** The Application Load Balancer routes standard backend web requests to the **Nginx** sidecar container, which reverse proxies to the **Odoo 19** container within the same task.
* **[ Odoo 19 Service ]** Deployed using the Odoo 19 community Docker image via native **ECS on EC2 (`c6g.xlarge` nodes)**, alongside an **Nginx sidecar**. It receives forwarded webhook payloads from the FastAPI agent via JSON-RPC, evaluates risk, gates AI tasks, binds directly to localized mount points referencing shared Amazon EFS file system for web filestores, and routes database traffic directly through the PgBouncer cluster discovery configuration. Valkey is used to store session data.
* **PgBouncer:** Runs on **ECS EC2 (`m6g.medium`)** nodes to handle dedicated lightweight database connection pooling.
* **AI Agent Backend (FastAPI + LangGraph):** Runs on **ECS EC2 (`m6g.large`)** nodes. It polls the **Amazon SQS** queue, forwards webhook payloads to Odoo via JSON-RPC, runs the LangGraph AI agent workflows when triggered by Odoo, and writes results back to Odoo via JSON-RPC. Deployed on dedicated node sizes to absorb memory spikes without impacting core database connectivity.


* **[ ElastiCache Serverless (Valkey) ]** Fully managed serverless caching engine used for instant sub-millisecond data retention (shared for Odoo session storage and agent state graph checkpointing). AWS manages scaling, high availability, and persistence natively without requiring manual AOF/EFS configurations.
* **[ Agent State & Checkpointing ]** LangGraph leverages **Amazon ElastiCache Serverless (Valkey)** to dynamically write, serialize, and read execution state graphs between automated node cycles or extended async breaks.
* **[ Telemetry Layer (Langfuse & ClickHouse) ]** Completely self-hosted natively on AWS to bypass external monthly subscription services:
* **Langfuse Server:** Deployed on **ECS Fargate Spot** tasks configured for strict cost optimization. The FastAPI agent buffers telemetry metrics into **Valkey** first, ensuring any brief Spot reclamation interruptions of Langfuse do not drop traces or delay upstream agent logic. Transactional configuration values write directly into a dedicated **RDS Postgres** instance (`db.t4g.small`).
* **ClickHouse Server:** Runs on **ECS EC2 (`r6g.xlarge`)** nodes as an isolated analytical container layer. It utilizes the native Amazon EBS integration for ECS tasks to manage volume lifecycle automatically, and uses a **Tiered Storage Architecture**, maintaining recent, high-value data on high-throughput Amazon **io2 EBS volumes** for fast querying, while automatically offloading older, historical trace dumps to **Amazon S3** via the free **Gateway VPC Endpoint** to significantly reduce long-term storage costs.


* **[ External API Egress ]** External API calls (Anthropic Claude, Shopify GraphQL, Slack REST) route through the Regional NAT Gateway. Secure runtime authentication hooks via AWS Secrets Manager handle credentials for external endpoints.

### Workflow Mapping to Architecture

**Path 1: Inventory Reconciliation (AI ops)**

1. An administrator triggers an action in **Odoo** for missing/mismatch stock, which then initiates an AI task via REST API to the **FastAPI + LangGraph container**.
2. The task starts natively inside the FastAPI `m6g.large` EC2 private pool.
3. Odoo fetches active sales logs via the **Shopify API** through the NAT Gateway and passes this context to the agent.
4. The agent interacts with Odoo via local JSON-RPC calls routed through **ECS Service Connect** to query further internal inventory moves.


5. Claude processes the variations via the Anthropic API, calculates the correct stock levels, and drafts a proposed synchronization fix.
6. [ Human Validation ] The agent pauses execution, saves its state to ElastiCache, and sends a proposed reconciliation summary via the Slack API to an administrator for review.
7. Upon receiving approval via a Slack interactive webhook, the agent resumes execution, updates Odoo's catalog records, and buffers execution steps, metrics, and parameters into ElastiCache, where a background worker asynchronously flushes them to Langfuse.

**Path 2: Fraud Detection (Autonomous Loop)**

Shopify fraud analysis is asynchronous, so this path is driven by two Shopify webhooks routed through the same Lambda -> SQS ingress: `orders/create` (order placed) and `orders/risk_assessment_changed` (fraud verdict, available once analysis completes).

0. When an order is placed, the `orders/create` webhook is ingested and forwarded to **Odoo**, which builds a **confirmed `sale.order`** (customer and line items mapped, full raw payload stored). Orders live in Odoo directly - no separate Shopify connector is required.
1. When Shopify's fraud analysis completes, the `orders/risk_assessment_changed` webhook triggers a payload to **AWS API Gateway**, which invokes the **Webhook Verify+Ingest Lambda** (proxy integration). The Lambda verifies the HMAC signature and, on success, enqueues it into the **Amazon SQS** queue.
2. **FastAPI SQS workers** poll the SQS queue privately via the VPC Interface Endpoint, retrieve the webhook payload, and forward it to **Odoo** via a private JSON-RPC call. Odoo correlates the verdict to the imported `sale.order` (the risk payload has no total, so the order total is recovered from it). If the order is very cheap (< $10) and marked with medium or high risk, Odoo automatically rejects/cancels the order in **both Shopify and Odoo** directly (without wasting AI compute). Otherwise, Odoo initiates an AI task back to the **FastAPI + LangGraph container** via a REST API call.
3. The task initiates inside a LangGraph execution thread. If Shopify flagged the order as **Medium Risk**, the agent invokes **Claude Haiku** for a fast, low-cost pre-screening. If flagged as **High Risk**, it invokes **Claude Sonnet** for deep context gathering (e.g., cross-referencing IPs, validating billing/shipping mismatches).
4. The agent pushes an interactive Block Kit card via the Slack API to the store manager. This card details the AI's fraud analysis and provides explicit [ Approve Order ] or [ Reject & Cancel ] action triggers.
5. The active execution state freezes immediately; the thread safely stores its state machine details inside **ElastiCache Serverless** and terminates the active thread to save compute resources.
6. A manager confirms or rejects the prompt inside the Slack interface. This passes a validation webhook response to **AWS API Gateway** and into the **SQS** queue. FastAPI polls this queue, seamlessly resumes the workflow from ElastiCache, and returns the final decision to Odoo via JSON-RPC. Odoo then executes the final decision (e.g., canceling the order in Shopify via the Shopify API and syncing the status).
