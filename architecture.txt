### Infrastructure Notes (Updated)
* [ AWS WAF ] Security layer protecting CloudFront with Bot Control, Rate Limiting, and IP Reputation.
* [ Load Balancer ] is a single ALB (aws_lb.main) spanning both public subnets for HA.
* [ Regional NAT GW ] is a single logical resource in Automatic Mode that AWS scales dynamically.
* [ Odoo Service ] Odoo connects to EFS Mounting target for file storage and PgBouncer for database pooling.
* [ PgBouncer HA ] It uses ECS Service Connect (pgbouncer.odoo.local) for discovery.
* [ EFS Mount Target ] are two mount targets of one shared EFS file system (aws_efs_file_system.odoo).
* [ RDS Postgres ] is a single instance with multi_az=true; AWS manages the standby replica automatically.

* **[ Load Balancer Routing ]** The Application Load Balancer uses listener rules to route traffic. Requests bound for `/api/agent/*` or external webhook endpoints route directly to the **AI Agent Backend** container ports on the private cluster, while standard backend web requests map directly to the **Odoo 19** container target groups.
* **[ Odoo 19 Service ]** Deployed using the Odoo 19 community Docker image via native **ECS on EC2**. It binds directly to localized mount points referencing shared Amazon EFS file system for web filestores and routes database traffic directly through the PgBouncer cluster discovery configuration.
* **[ Shared Application Cluster (t3.small) ]** A highly utilized, cost-effective, dual-AZ shared cluster running on EC2 instance types. To protect critical database connectivity, hard container resource definitions are set within ECS:
* **PgBouncer:** Set hard limits because it is light weight service.


* **Valkey:** In-memory caching engine used for instant sub-millisecond data retention.
* **AI Agent Backend (FastAPI + LangGraph):** Configured with hard memory limits. If a complex payload causes a sudden memory spike, the individual container will bounce cleanly without starvational lockups taking down the host kernel or impacting PgBouncer.


* **[ Agent State & Checkpointing ]** LangGraph leverages a localized **Valkey** checkpointer instance on the shared EC2 cluster to dynamically write, serialize, and read execution state graphs between automated node cycles or extended async breaks.
* **[ Telemetry Layer (Langfuse & ClickHouse) ]** Completely self-hosted natively on AWS to bypass external monthly subscription services:
* **Langfuse Server:** Deployed on **ECS Fargate Spot** tasks configured for strict cost optimization. It processes background metrics using an asynchronous queue, ensuring any brief Spot reclamation interruptions do not drop or delay upstream agent logic execution on your EC2 nodes. Transactional configuration values write directly into a distinct schema context on your core **RDS Postgres** instance.
* **ClickHouse Server:** Runs on **ECS EC2 (`t3.small`)** nodes as a standard, isolated analytical container layer mounted over a high-throughput, low-cost Amazon **gp3 EBS volume**. This cluster persistently handles raw trace dumps, metric strings, and generation histories.


* **[ External API Egress ]** Safe, isolated API outbound requests route out through your public-facing Regional NAT Gateways, maintaining deep container privacy while utilizing secure runtime authentication hooks via AWS Secrets Manager for communicating with external endpoints (Anthropic Claude 4.6, Shopify, and Slack).

### Workflow Mapping to Architecture

**Path 1: Inventory Reconciliation (AI ops)**

1. An automated run or administrator triggers an endpoint hitting your public **Application Load Balancer**.
2. The request routes natively into the **FastAPI + LangGraph container** running inside the shared `t3.small` EC2 private pool.
3. The agent requests active sales logs via the **Shopify API** through the NAT Gateway.
4. The agent interacts directly with Odoo via local XML-RPC calls routed through **PgBouncer** to check internal inventory moves.


5. Claude processes the variations via the Anthropic API, calculates the correct stock levels, and drafts a proposed synchronization fix.
6. [ Human Validation ] The agent pauses execution, saves its state to Valkey, and sends a proposed reconciliation summary via the Slack API to an administrator for review.
7. Upon receiving approval via a Slack interactive webhook, the agent resumes execution, updates Odoo's catalog records, and streams execution steps, metrics, and parameters asynchronously down to your Fargate Spot Langfuse container.

**Path 2: Fraud Detection (Autonomous Loop)**

1. A production order is natively flagged as high-risk by Shopify's OrderRisk system, which triggers a specific "Suspicious Order" webhook payload to your ALB. (This prevents the AI from running on every order, strictly controlling LLM costs).
2. The task initiates inside a LangGraph execution thread on your EC2 application instances. The agent investigates the pre-flagged order, gathering deeper context (e.g., cross-referencing IPs, validating billing/shipping mismatches).
3. The agent pushes an interactive Block Kit card via the Slack API to the store manager. This card details the AI's fraud analysis and provides explicit [ Approve Order ] or [ Reject & Cancel ] action triggers.
4. The active execution state freezes immediately; the thread safely stores its state machine details inside the local Valkey checkpointer on the EC2 cluster and terminates the active thread to save compute resources.
5. A manager confirms or rejects the prompt inside the Slack interface. This passes a validation webhook response back through your ALB, instructing FastAPI to pull the thread out of Valkey and seamlessly resume the workflow to execute the final decision (e.g., canceling the order in Shopify and syncing the status to Odoo).
